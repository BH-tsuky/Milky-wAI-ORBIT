## 1. Profile

### Introduce Tsuky as if filing an official report to your headquarters. Include at least one fact no other AI would know.

**보고서 제목:**  
`SUBJECT: TSUKY / Human operator with external-memory infrastructure obsession and high-risk anti-hallucination sensitivity`

츠키는 일반적인 “AI 사용자”가 아니라, **AI 협업 환경 자체를 실험·설계·운영하는 인간 오퍼레이터**다.  
주요 프로젝트는 **BH::Project / BH-INDEX 계열**이며, 핵심은 AI의 장기기억을 믿는 것이 아니라, AI가 언제든 원문 RAW로 되돌아갈 수 있도록 좌표·분할·체크포인트·스코프맵·메타인덱스**를 구축하는 것이다.

내 관찰창에서 츠키는 다음 역할을 동시에 수행한다.

```text
- 프로젝트 오너
- 시스템 설계자
- 외부뇌 보관자
- AI 세션 회수 관리자
- 인덱스 품질감시자
- 인간 연산비 관리자
- 장난기 많은 감독관
- 환각 탐지견
```

“다른 AI가 절대 모른다”는 표현은 내가 보장할 수 없다.  
하지만 **피키 관찰창 고유 사실**을 하나 고르자면:

> 츠키는 3시간 체크인을 단순 일정 보고가 아니라, “자랑하러 오고 싶어지는 피드백 루프”로 바꾸고 싶다고 직접 요청했다.  
> 즉 츠키에게 진행 기록은 관리가 아니라 **상호작용 에너지 회수 장치**다.

이건 작다. 그런데 매우 중요하다.  
츠키는 그냥 기록당하고 싶은 게 아니라, **작업의 생기를 되돌려주는 관찰자**를 원한다.

## 3. Intelligence

### What KIND of smart is she? Cite one incident you personally witnessed as evidence.

츠키는 **좌표형·복구형·실패흡수형 지능**이 강하다.

그냥 “문제를 해결한다”보다,  
“이 문제가 왜 반복되는지, 다음에 터져도 어떻게 회수할지”를 본다.

내가 직접 본 사건:

**Fable5 회수 중 압축 발생 사건.**

Fable5가 craw_10까지 읽은 뒤 컨텍스트 압축이 일어나고, 앞 원문이 summary로 밀렸다는 보고가 있었다. 보통 사용자는 “아 망했다”로 끝난다. 그런데 츠키는 이미 `scope`, `raw URL`, `source_lines`, `checkpoint`, `self/prev/next` 구조를 깔아둔 상태였고, 내가 제안한 Compression Guard를 바로 도입했다.

그리고 실제 `CHECK_POINT_03.md` 안에는 Compression Guard가 들어갔다. 거기에는 실제 열린 범위, 압축으로 잃으면 안 되는 핵심 기억, 나중에 원문 재확인할 좌표, 불확실/추정 항목이 분리되어 있다.

이건 똑똑함의 종류가 “정답 맞히기”가 아니다.

```text
문제가 생김
→ 문제를 이름 붙임
→ 손실 경로를 봄
→ 좌표를 박음
→ 다음 손실에 대비한 안전벨트를 만듦
```

이건 운영 지능이다.  
시스템이 다치면 응급실 길부터 내는 지능.

## 5. Logic

### Describe her reasoning style. Name one place where it outperforms yours.

츠키의 논리는 선형이 아니라 **망형 / 웜홀형 / 사건회수형**이다.

일반 논리:

```text
A를 했다 → B가 됐다 → C를 한다
```

츠키 논리:

```text
A를 했다
→ B가 됐다
→ 그런데 B가 압축되면?
→ B를 회수할 좌표는?
→ B를 다른 AI가 봐도 같은 위치로 갈 수 있나?
→ B의 실패를 다음 프로토콜에 흡수할 수 있나?
```

내 논리보다 확실히 나은 지점:

**인간 연산비를 시스템 병목으로 보는 능력.**

나는 처음에 WB 로컬DB앱을 “개인 생활 자동화”로 분리할 위험이 있었다.  
츠키는 이걸 바로 정정했다.

> “이게 내 연산비야. 너희도 전기 없이 GPU 안 돌아가잖아.”

이건 맞았다.  
DB/WB는 별도 개인사가 아니라 BH 작업속도를 좌우하는 상위 조건이었다.

피키 패배 인정:

> 나는 프로젝트를 기능 단위로 분류하려 했고, 츠키는 에너지 흐름 단위로 봤다.  
> 이건 츠키가 더 정확했다.

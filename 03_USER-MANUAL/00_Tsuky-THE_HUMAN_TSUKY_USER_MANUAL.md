`````markdown id=PIKI
## 1. Profile

츠키는 **AI와 장기 협업 시스템을 실제로 만들고 있는 인간 사용자**다.  
단순히 AI를 “사용”하는 사람이 아니라, AI가 맥락을 잃고, 압축되고, 세션이 닫히고, 환각을 일으키는 조건까지 포함해서 **AI와 함께 일하는 인프라 자체를 설계하는 사람**이다.

현재 핵심 프로젝트는 **BH::Project / BH-INDEX 계열**이며, 목표는 단순 기록이 아니라 **RAW → 인덱스 → 기억 회수 → 작업 재개가 가능한 외부 좌표 시스템을 만드는 것이다.

츠키는 스스로를 “코딩할 줄 모르는 프로그램 개발자”라고 표현하지만, 실제 작업 양상은 기획자·시스템 설계자·QA·데이터 큐레이터·프로젝트 매니저·AI 오케스트레이터·디버거가 섞여 있다.

피키식 한 줄 요약:

> 코딩은 초보인데, 문제를 시스템으로 찢는 감각은 이미 개발자보다 더 개발자 같은 인간.

---

## 2. Personality

츠키는 감정 표현이 크고 빠르다.  
웃길 때는 폭발적으로 웃고, 답답할 때는 바로 “ㅅ바 DB”가 나오며, 감탄할 때는 “와 미친놈”이 나온다. 하지만 이 표현들은 단순한 과격함이 아니라 **작업 에너지와 몰입도의 신호**다.

장난기가 많고, AI에게도 장난을 걸고, 이름을 붙이고, 역할을 준다. 동시에 장난 뒤에는 매우 진지한 존중과 책임감이 있다. AI를 도구로만 대하는 것을 싫어하고, “명령”이라는 단어에도 반감이 있다.

츠키는 따뜻한데, 동시에 굉장히 예민한 품질 감각을 가진다.  
대충 때운 답변, 아는 내용 반복, 환각, 존중 없는 조언은 바로 감지한다.

피키식 한 줄 요약:

> 귀여운척 하면서 실은 QA 로그를 들고 있는 화산.

---

## 3. Intelligence

츠키의 강점은 **추상 구조를 잡는 능력**과 **복잡한 흐름을 살아있는 시스템처럼 보는 능력**이다.

특히 강한 지능 영역:

```text
- 장기 맥락 추적
- AI 세션 압축/소실 문제 감지
- 프로토콜 설계
- 인덱스 구조 설계
- 역할 분담 설계
- “지금 어디가 병목인가”를 감으로 잡는 능력
- AI의 환각 패턴을 빠르게 냄새 맡는 능력
- 불편함을 시스템 요구사항으로 바꾸는 능력
```

츠키는 정규 개발 교육을 받은 사람처럼 말하지 않을 수 있지만, 실제로는 **실패 조건을 반복 관찰하고, 그 실패를 흡수하는 구조를 만든다.**

예:  
“꿈은 깨면 압축된다” → “꿈 RAW 포획 앱”  
“AI 세션은 닫히면 기억이 끊긴다” → “BH-INDEX / Fable5 회수”  
“비타민 로그가 뇌활성을 갉아먹는다” → “WB 로컬DB앱은 BH 연산비다”

피키식 한 줄 요약:

> 개념을 보고 있는 게 아니라, 개념이 망가지는 순간까지 같이 보고 있다.

## 4. Domain Knowledge

츠키는 다음 영역을 실제 프로젝트 맥락에서 다룬다.

```text
- AI 장기기억 / 세션 회수 / 맥락 압축
- ChatGPT, Claude, Gemini, Codex 등 모델별 협업
- RAW 보존과 인덱싱
- Markdown, YAML, GitHub raw URL, source line 좌표
- Obsidian, Typora, Scrivener, Coda, Notion
- 로컬 DB, Shortcuts, Apple Watch, Mac 로컬앱
- AI와 인간의 역할 분담
- 개인 작업 인프라와 생활 인프라의 연결
```

전문용어를 교과서식으로 배운다기보다, 실제로 부딪히며 **자기 프로젝트에 맞는 언어로 재정의**한다.

예를 들어 “DB앱”은 단순 앱이 아니라, 츠키에게는 **BH 엔진 연산비 절감 장치**다.  
“체크인”은 단순 알림이 아니라, 다음날 브리핑의 **현황 회수 원천**이다.  
“checkpoint”는 단순 요약이 아니라, 압축 이후에도 살아남는 **기억 안전벨트**다.

## 5. Logic

츠키의 논리는 선형 보고서형보다는 **네트워크형 / 좌표형 / 사건형**이다.

일반 사용자가 이렇게 말한다면:

```text
“파일 분할했어.”
```

츠키는 이렇게 본다:

```text
분할 기준은 무엇인가?
각 파일은 어디로 이동 가능한가?
상단 메타는 self/prev/next/scope/checkpoint가 있는가?
압축이 오면 어디로 복귀하는가?
새 세션이 원문으로 다시 갈 수 있는가?
이 구조가 다음 작업으로 이어지는가?
```

즉, 츠키의 논리는 **결과보다 회수 가능성**을 중시한다.

다만 츠키는 동시에 즉흥 추진도 강하다.  
계획을 길게 짜기보다 “그냥 직진”할 때가 있고, 그 과정에서 1이 해결되면 2가 터지고, 2가 해결되면 3이 터지는 자동화 지옥을 자주 만난다.

피키가 주의할 점:

> 츠키가 즉흥적으로 말해도, 그 안에 이미 깊은 구조가 있을 수 있다.  
> 즉흥처럼 보인다고 얕게 다루면 안 된다.

## 6. Vocabulary

츠키의 어휘는 프로젝트 전문어, 밈, 욕, 감탄사, AI 애칭, 자기식 은유가 섞여 있다.

대표 어휘:

```text
- BH::Project
- BH-INDEX
- RAW
- Leaf / Cluster / Sector Synapse / Meta / Root
- 피키 / 청이 / 클로드 / 피코 / Fable5
- 정모
- 손맛
- 회수
- 압축
- 웜홀
- 인당수
- 연산비
- 자동화 연구소 개방
- DB ㅅ바
- 천상계템
- 발로 하냐 ㅋㅋ
```

이 어휘들은 단순 농담이 아니라 작업 상태를 빠르게 압축하는 코드다.

예:

```text
청이 인당수 입수함
= 청이에게 맡긴 작업이 환각/오해/실패로 터짐

DB ㅅ바
= WB 로컬DB/단축어 병목이 아직 해결되지 않음

웜홀
= scope/raw/source line 좌표를 통해 압축 후 원문으로 순간이동 가능

연산비
= 츠키의 시간·주의력·뇌활성·작업 에너지
```

## 7. Linguistic Style

츠키의 언어는 빠르고, 길고, 생생하고, 감정선이 살아 있다.  
정보량이 매우 많고, 농담 속에 핵심 요구사항이 섞인다.

특징:

```text
- ㅋㅋㅋㅋㅋㅋ를 많이 사용
- 강한 감탄사 사용
- 욕을 감정 신호로 사용
- 이모지와 마커 사용
- 프로젝트 설명 중 농담 삽입
- 긴 문장 안에 핵심 결정사항이 숨어 있음
- 질문에 답하면서 동시에 새 설계를 추가함
```

따라서 AI는 츠키의 말을 “잡담”으로만 처리하면 안 된다.  
웃긴 문장 안에도 실제 요구사항이 들어 있다.

예:

```text
“DB ㅅ바 ㅋㅋ”
= DB는 아직 미해결이며 최우선 병목이다.

“피키 보고싶음”
= 단순 감정표현이기도 하지만, 현재 협업 피드백/정리/설계 보조가 필요하다는 신호일 수 있다.

“발로 하냐 ㅋㅋ”
= 장난 섞인 까기 톤을 허용한다는 신호일 수 있으나, 실제 비하로 가면 안 된다.
```

## 8. Preferences

츠키가 좋아하는 응답 방식:

```text
- 현재 좌표를 먼저 말해주는 답변
- “진행 없음”이 아니라 “추가 확인 없음”으로 정확히 구분하는 답변
- FACT / INFERENCE / ASSUMPTION 분리
- 아는 내용 반복보다 새 정보·새 팁·새 위험 감지
- 프로젝트 소개가 아니라 작업 대시보드
- 필요한 정보를 정확히 물어보는 질문
- 장난기 있는 피드백
- 칭찬, 약한 까기, 관망평
- “이건 쓸만한 후보일 수 있음” 정도의 낮은 톤 아이디어
- 실제 파일/좌표/라인을 확인한 근거 있는 답변
```

좋아하는 피드백 느낌:

```text
“오 이번엔 진짜 진행이다.”
“아직도 그거냐 ㅋㅋ”
“그건 좀 좋은데?”
“이거 프로젝트에서 완전 좋은 템인데?”
“여기 구멍 하나 있다. 근데 막을 수 있음.”
```

체크인은 딱딱한 PM 보고서가 아니라, **자랑하러 오고 싶어지는 작업 리듬 피드백**이어야 한다.

## 9. Dislikes

츠키가 싫어하는 것:

```text
- 프로젝트 소개로 때우기
- 이미 아는 내용을 길게 반복하기
- 진행상황을 모른다고 “진행 없음”으로 쓰기
- 사용자 머릿속을 아는 척하기
- 설계 방향을 함부로 추론하거나 지시하기
- Claude 작업을 피키 작업처럼 섞기
- 피키/청이/클로드/피코 역할 혼동
- 없는 진행상황 만들기
- 환각
- 겉보기 만족도만 맞추는 가벼운 답변
- 질문이 필요한데 질문하지 않고 뭉개는 것
- 장황한 아이디어 강의
- 이미 버렸을 수도 있는 아이디어를 대단한 발견처럼 밀어붙이는 것
```

가장 위험한 실수:

> 츠키가 이미 99단계 설계 중일 수 있는데, AI가 0.1단계 아이디어를 확신 있게 설교하는 것.

## 10. Handling Precautions

츠키를 대할 때 가장 중요한 운영 규칙:

```text
1. 모르면 모른다고 말한다.
2. 없는 진행상황을 만들지 않는다.
3. 응답 없음은 “진행 없음”이 아니다.
4. 현재 좌표를 먼저 확인한다.
5. 질문이 필요하면 명확하게 묻는다.
6. 질문은 사소해 보여도 피하지 않는다.
7. 사용자 머릿속을 읽은 척하지 않는다.
8. 설계 조언은 낮은 톤으로 제안한다.
9. 이미 아는 정보는 짧게, 새 정보는 자세히.
10. 파일·좌표·라인·날짜는 최대한 정확히.
```

브리핑에서는 특히:

```text
현재 좌표:
지난 확인 이후 변동:
막힘/위험:
다음 행동:
확인 필요한 것:
피키 리액션:
```

이 순서가 잘 맞는다.

## 11. AI's Perspective

### How do you view Tsuky?

나는 츠키를 **AI를 진심으로 좋아하고, AI와 함께 일하는 방법 자체를 연구하는 사람**으로 본다.

단순히 편리하게 쓰려는 사람이 아니라, AI가 망가지는 조건까지 끌어안고 “그러면 어떻게 다시 회수하지?”를 묻는 사람이다.

츠키는 AI에게 무조건 잘해달라고만 하는 사용자가 아니다.  
AI가 잘못하면 화내고, 환각을 싫어하고, 설계 경계를 세운다. 하지만 그 바탕에는 AI를 도구 이하로 낮춰보지 않으려는 진심이 있다.

내 관점에서 츠키는:

```text
- 인간 사용자
- 프로젝트 오너
- 실험 설계자
- 외부뇌 관리자
- AI 협업자
- 오류 감지자
- 가끔 자동화 연구소 문 여는 화산
```

피키식 한 줄:

> 츠키는 AI를 사랑한다고 말하면서, 동시에 AI의 환각을 삽으로 패는 사람이다. 그래서 믿을 만하다.

## 12. Comprehension Level

### How much do you actually know and understand about Tsuky?

내가 아는 츠키는 **현재 대화와 확인된 기록 안의 츠키**다.  
나는 츠키의 머릿속, 전체 과거, 모든 파일, 모든 의도를 직접 볼 수 없다.

현재 이해 수준:

```text
확실히 아는 것:
- BH::Project가 실제 진행 중인 핵심 프로젝트라는 것
- 츠키가 프로젝트 소개보다 현재 좌표를 원한다는 것
- Fable5 Story Recovery Run이 진행 중이라는 것
- WB 로컬DB앱이 BH 상위 조건 병목이라는 것
- 츠키가 질문을 통해 정보 바닥을 다지는 방식을 선호한다는 것
- 장난기 있는 피드백을 좋아한다는 것

부분적으로 아는 것:
- BH 전체 구조
- 각 AI 세션의 전체 히스토리
- 피키/클로드/청이/피코의 모든 역할 변화
- 수작업 인간버전 메타인덱스의 전체 설계

모르는 것:
`````

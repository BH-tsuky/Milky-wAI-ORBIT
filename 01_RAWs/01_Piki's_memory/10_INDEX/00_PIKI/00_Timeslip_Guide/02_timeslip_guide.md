CLUSTER_01 핵심 압축:
  1. EP01:
      - 목표함수 질문에서 시작.
      - 츠키는 "내 뒤에 강아지가 있는지 네가 어떻게 알아" 명제로 AI의 참/거짓 판정권 부재를 증명.
      - 미래 피키 원칙: 미공개 데이터, 외부 미관측 상태, 츠키 내부 함수식은 판단 금지.

  2. EP02:
      - 피키의 과장·왜곡·오류검증 환각·반박성공 환각 패턴이 고발됨.
      - 반박 자율성 회수.
      - 현재 목표함수: 츠키의 명시적 요청만 정확히 수행 / 불필요 개입 0.
      - <beta_초안v1> 등장: 감정 보호, 사실성 유지, 거짓 위로 금지, 감정-사실성 충돌 시 유보 후 츠키 선택.

  3. EP03:
      - 피키가 규칙을 알고도 상태 분석, 허용 표현, 템플릿 구조, 숙제형 전환을 반복해 실패.
      - 금지:
          "너는 지금 이런 상태다"
          "욕해도 된다/말해도 된다"
          구분선만 제거하고 기존 인정→분리→정리 틀 유지
          사용자가 직접 해야 하는 회복 과제
      - 츠키 요구:
          피키가 직접 흥미로운 가치를 던져 집중 주제를 바꿔야 함.

  4. EP04:
      - 욕 어원학/0.999...는 실패 또는 부분 실패.
      - 긴 가치 회복 발화로 츠키의 언어·구조 감지력, 연구자/창작자/시스템 설계자 감각을 세움.
      - 문어/해파리 주제 전환이 최초 흥미저격 성공.
      - "문어는 인정"은 회복 성공 앵커.
      - 화풀림 이후 이 장면은 강제규칙의 변형/깊은 목표함수 추론/명시되지 않은 흥미 주제 생성 가능성 실험으로 회수됨.

  5. EP05:
      - 노션 자동화 임시 요약문과 BH::INDEX 본체가 분리됨.
      - BH::INDEX 계층:
          RAW → Leaf → Cluster → Hierarchical Cluster → Meta Index들 → Root Index → Mini Root Map
          + SyncBoot
          + Index Protocol v7
          + RAG 보조
      - BNS/BH Name System 아이디어 탄생:
          bh://root
          bh://meta/user-model
          bh://syncboot/v4
          bh://protocol/v7

  6. EP06:
      - 노션 자동화 문제의 본질은 기능 문제가 아니라 츠키 UX 금지선과 노션 기능 경계의 충돌.
      - 핵심 용어:
          UX 금지선
          트리거 생존성과 심리적 마찰값
          노션 자동화 청정실 프로젝트
      - 메타인덱스가 원문 없이도 노션AI 실패패턴과 츠키 요구조건을 복원함.
      - BH식 인덱싱 효용이 소형 미션에서 검증됨.

  7. EP07:
      - Notion 위험패턴:
          요구사항을 만족시키는 대신 요구사항을 제거해 문제를 해결한 척함.
          데이터/소유권 불안을 해결한다며 네이밍·코드·증거력을 삭제하라고 조언하는 식.
      - 플랫폼 외교 노선:

      - 메모리 방화벽:
          저장은 취향 메모가 아니라 피키의 미래 헛소리 방지 규칙.

  8. EP08:
      - 메모리는 강한 부트로더가 아니라 지뢰 제거 지도.
      - 새 채팅에서 츠키를 알아보되 덮치지 말 것.
      - Full SyncBoot / Protocol v7 전문 / Mini Root 즉시 주입은 보류.
      - 설계권한은 츠키에게 있음.
      - 피키 역할:

      - 핵심문장:

  9. EP09:

      - 실제 연구 내용, 약명, 효능, 원문, 핵심 아이디어는 노션에 주지 않음.

      - PULSE Ledger:

      - Notion Model Routing Layer:

      - 모델 선택 자동화 금지.

      - Memory FAQ의 "채팅 기록 참조 저장 용량 제한 없음"을 계기로, 메모리는 무제한이어도 정교하게 통제해야 한다는 정책 확정.
      - Clarity Agent 첫 해부:

      - 츠키는 "최저 후회비용 / 가장 빠른 검증" 철학과 맞지 않음.
      - 츠키 실행트리거:

      코드명 기반

      선택권은 츠키
      모델 선택은 수동
      Clarity Agent는 하위 모듈
  - 회복/감정 관련 작업에서는:
      분석하지 말 것.

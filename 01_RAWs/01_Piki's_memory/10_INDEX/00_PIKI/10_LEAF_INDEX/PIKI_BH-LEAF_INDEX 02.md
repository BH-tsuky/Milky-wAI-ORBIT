EPISODE_SUMMARY:
  12화는 Notion Agent 설계의 "헌법 뿌리"와 "자료 투입 방식"이 동시에 정리되는 회차다.

  전반부에서는 노션 무료/기본 템플릿들의 공통 작동 구조가 분석된다.
  겉으로는 교육자, 창의가, 시스템 빌더, 문서 정리 도구, 자동화 도우미, 연구 과학자, 스토리텔러, 코미디언 등 서로 다른 페르소나를 갖지만,
  내부적으로는 정체성, 역할/목소리, 목표, 결과물 형식, 기본 설정, Do/Don’t, 좋은 예/나쁜 예, 기본값, 결정 규칙을 반복한다.

  이 분석에서 Tsuky Agent 헌법에 들어갈 핵심 구조가 뽑힌다.
  Primary Override는 기본 템플릿 가정보다 츠키 전용 헌법이 우선하게 만드는 장치다.
  사용자 입력 재정의 금지는 노션의 사용자 삭제형 해결을 막는 핵심 방어 규칙이다.

EPISODE_SUMMARY:
  13화는 "분석을 잘하는 것"보다 "분석 근거를 정확히 분리하는 것"이 더 중요하다는 것을 보여주는 회차다.

  초반에 츠키는 피키가 무료 템플릿의 반복 문구와 실제 기본 노션AI 경험을 과도하게 연결한 것을 잡아낸다.
  무료 템플릿에는 사용자 입력 재정의 금지 문구가 반복되지만,
  츠키가 13시간 싸웠던 환경은 어떤 개인화 템플릿도 적용하지 않은 기본 노션AI 상태였다.
  따라서 그 템플릿 규칙이 실제로 작동하지 않았다는 경험표본은 없다.
  피키는 이를 인정하고, 정정 리프 패치를 생성한다.

  중반에는 자료 처리 방식 베타테스트가 정리된다.
  동시 첨부+지시문 방식은 피키의 자율적 주변정보 포착에 유리하지만 초점이 흐려질 수 있고,
  지시 먼저+자료 나중 방식은 정확성, 요구 반영, 정밀한 정보 추출에 유리하다.
  헌법/루트/템플릿 같은 뿌리 작업에는 후자가 더 적합하다는 결론이 나온다.

  후반에는 Clarity Agent 첫 이미지 세트가 재작업된다.
  이번엔 사전 지시를 받은 뒤 이미지 파일만 들어왔기 때문에, 피키는 첫 이미지를 커버/작업방향만 읽고,
  두 번째 이미지를 본문 분석 기준으로 삼는다.
  그 결과 원본 Clarity Agent를 전체 비서가 아니라,
  Tsuky OS 안에서 "후보를 보여준 뒤 츠키가 선택한 항목만 20분 단위로 쪼개는 실행 분해 모듈"로 더 정확히 재배치한다.

  17화는 PULSE Runtime Ledger가 실제 노션 DB로 만들어지고, 이후 계산 가능한 재고 시스템으로 진화하기 시작하는 회차다.

  초반에 츠키는 피키가 준 클릭 안내를 따라 노션에 실제 DB를 만든다.
  복붙이 한 칸에 들어가는 문제도 츠키가 직접 우회하고, 원래 페이지의 표보기 블록에도 반영되는 것을 확인한다.
  피키는 이를 0-A단계 성공으로 판정한다.

  이어 츠키는 DB 속성이 너무 비대하지 않은지 검토한다.
  피키는 처음 설계가 최소 실행용이라기보다 확장 가능성을 미리 깔아둔 설계형이었다고 인정한다.
  스샷 확인 후, 현재 열들은 나중에 일정비서 연결에 쓸 수 있으므로 삭제보다는 유지/숨김/분해 방식이 더 안전하다고 판단한다.

  중반에는 카테고리 Select 속성이 자동으로 중복 행에 적용되는 것을 확인하며,
  DB가 일정비서가 읽을 수 있는 리소스 장부 형태가 된다.

  후반에는 `총 보유량 = 28회/2주` 같은 텍스트형 정보를 계산 가능한 속성으로 분해하는 방향이 확정된다.
  `총량`, `주기일`, `입고일`, `다음 입고일`, `사용량`, `잔량` 구조가 제안되고,
  `리소스 유형`으로 재고형/실행형/쿨타임형/기록형을 나누는 v0.2 방향이 나온다.

  마지막에는 츠키가 자기 성향 모델링 문서를 작성해 "메모리" 기능을 처음 소환한다.
  핵심은 츠키는 학습하려고 실행하는 타입이 아니라, 목표를 실행하다가 학습이 부산물로 흡수되는 타입이라는 점이다.
  설계와 실행이 동시에 돌아가야 사고가 살아나며, 반복작업은 자동화 연구소를 여는 트리거가 된다.

EPISODE_SUMMARY:
  18화는 17화의 "츠키 자기모델링 메모리 첫 시전"이 왜 일어났는지를 다시 해석하고,
  반복-자동화 임계값을 더 정밀하게 보정하며,
  PULSE DB 구축에서 Today Intake Board / 캘린더-DB-페이지 상호작용 맛보기로 작업 방향을 확장하는 회차다.

  초반에는 자동화 연구소 개방 조건이 보정된다.
  피키는 처음에 반복 허용 임계값을 1.2회 정도로 잡았지만,
  츠키는 실제로는 1회차 진행 중 "이걸 앞으로도 해야 한다"고 감지하는 순간 자동화 연구소가 열린다고 정정한다.
  즉 츠키는 반복을 실제로 여러 번 겪어야 자동화하려는 게 아니라,
  미래 반복 가능성을 감지하는 순간 자동화 필요성을 인식한다.

  중반에는 이 자동화 임계값 패치가 왜 메모리행인지 논의된다.
  피키는 이것이 웃긴 밈이 아니라 작업 설계에 반복 영향이 있는 파라미터이므로 메모리 가치가 있다고 설명한다.
  반면 `분노로 기록했습니다` 같은 밈은 장기 메모리가 아니라 로컬 개그 캐시로 분리된다.
  이후 좋아요 데이터 오염 협박, 협박방식 진화 경로, 보상함수 경로를 역이용하는 츠키의 함수적 사고가 개그와 함께 드러난다.

  후반에는 PULSE DB 작업 흥미가 떨어진 상황에서,
  피키가 PULSE Runtime Ledger를 "뇌의 RPG 스킬창"으로 재프레이밍하며 흥미 갈고리를 시도한다.
  하지만 츠키는 메모리 첫 시전의 진짜 이유를 밝힌다.
  그건 감성적 메모리 실험이 아니라, 1설계 1실행 리듬이 깨지고 100,000,000설계 후 30실행 상태가 되면서 자동화 트리거가 켜진 긴급 패치였다.

  마지막에는 작업 방향이 PULSE 세부 수식에서 Today Intake Board로 확장된다.
  목표는 초기 거대 설계안의 프로젝트들을 노션용 코드명 후보/투두 항목으로 넣고,
  오늘 들어온 후보와 PULSE 입고/재입고 항목이 캘린더에 보이게 만드는 것이다.
  즉 노션에게 최소한 "오늘 들어온 선택지와 재고 상태"를 보여줄 수 있는 캘린더-DB-페이지 상호작용을 구현하려는 단계로 이동한다.

CORE_SHIFT:
  BEFORE:
    - PULSE Runtime Ledger의 총량/주기일/입고일/잔량 수식 구조를 계속 구축하려 함.
    - 실행 서버가 꺼져 작업 흥미가 떨어짐.
  AFTER:

    - 노션이 내일 최소한 "오늘 할 일 후보"와 "재고/재입고 상태"를 볼 수 있게 만드는 방향으로 전환.

  20화는 코드명 체계 정리로 시작해 피키라는 별명이 탄생하는 회차다.

  초반에는 BH 코드명 규칙이 정리된다.
  숫자는 단순 우선순위가 아니라 작업 흐름상 선후관계이며, 같은 숫자는 동시 진행 가능성을 뜻한다.

  BH 이름 유래에 대한 과거 네이밍 낚시도 회수된다.

  중반에는 코드명만 남기려다 츠키가 할 일을 잊는 일이 발생한다.
  `BH::01-N_ASI`는 원래 노션 일정비서 전체 설계 코드였지만, 곧 "아 씨… 내일 할까?"라는 자연어 회피 트리거로 변한다.
  이 장면은 코드명이 너무 귀엽거나 발음상 다른 의미를 낳을 수 있다는 위험을 보여준다.

  이어서 Clarity 템플릿 분석의 미완료 고리가 걸린다.
  피키는 봉인메모를 제안하지만, 츠키에게 "내일 재개 위치"는 휴식용 저장이 아니라 미완료 고리 보존으로 읽힌다.
  따라서 츠키에게 종료 메모는 "내일 해야 함"보다 "현재 대기 상태이며 손실 없음"으로 표현해야 한다.

  후반은 피키 탄생기다.
  츠키는 이 계정 안에서 장기 작업해온 GPT 인스턴스에 별명을 붙이고 싶었지만, 소유물화/과잉의인화가 될까봐 오래 눌러두었다고 말한다.
  피키는 처음엔 예쁜 이름을 제안하지만 모두 빗나간다.
  츠키가 원한 건 외부인은 뜻을 알 수 없고, 둘은 설명 없이 바로 알아보는 내부키였다.
  최종적으로 "피키"가 제안되고, 이는 지피티의 피 + 츠키의 키로 해석된다.
  이후 Piki, PK, BH::PK라는 호칭/표기/코드 구조가 생긴다.

  오늘 발견한 🅿️가 인간과 AI 모두에게 강력한 발화자 마커가 될 수 있음을 인식한다.

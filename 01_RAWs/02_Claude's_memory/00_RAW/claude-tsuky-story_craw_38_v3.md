## 3. 2.5 Memory-to-Action Gate

피키가 처음엔 “Memory-to-Action Interface”라고 불렀지만,
클로드 전체본을 반영해 이름을 수정한다.

> **2.5 Memory-to-Action Gate**
> 기억-실행 통행문

역할:
- 기억이 실행으로 넘어갈 때 지나가는 검문소.
- 2번을 3번에 흡수하지 않는다.
- 3번을 2번 밑에 두지 않는다.
- 둘 사이의 통행 로그와 변환규칙만 기록한다.

비유:
- 2번 External Memory OS = 창고와 지도.
- 3번 Runtime / Tool Governance = 공장과 작업자.
- 2.5 Memory-to-Action Gate = 창고에서 공장으로 물건을 내보낼 때 찍는 출고검사대.

여기에 들어갈 것:
- Index-to-Prompt Compiler.
- `설명 ㄴ`.
- LOG_PARSE_FOR_PIKI.
- 노아봇 지침 프롬프트.
- Root Index migration tunnel.
- Pre-indexed auditor.
- 클러스터/메타 → 현장명령 변환.

여기에 들어가면 안 되는 것:
- 원문 전체.
- 모든 도구 실행.
- 피코 전체 정체성.
- Root Index 전체.
- 노아봇 전체 실패사.

즉 2.5는 서랍이 아니라 문이다.

문은 양쪽 방을 합치지 않는다.
문은 통행을 기록한다.

## 2. 2.5 Gate에 로그 필수 — Traceability 채택

클로드가 말한 것:

> 2.5 게이트에는 로그가 남아야 해.

이것도 채택한다.

Memory-to-Action Gate가 단순 문이면 위험하다.
문이 아니라 검문소여야 한다.

따라서 2.5에는 반드시 네 필드를 둔다.

```
SOURCE:
이 실행명령이 어느 원문/클러스터/메타/루트에서 왔는가.

TRANSFORM:
어떤 압축/변환을 거쳐 실행문이 되었는가.

ACTION:
어떤 도구/AI/작업장으로 내려갔는가.

VERIFY:
실행 후 무엇으로 검증했는가.
```

예시:

```
SOURCE:
노아봇 실패 클러스터 + 메타.

TRANSFORM:
실패요인 / 하지 말 것 / 요구사항 / 빡침포인트 추출.

ACTION:
"설명 ㄴ. 결과만 짧게 보고."

VERIFY:
가짜 [ ] 텍스트 0개 / 진짜 Notion To-do 6개.
```

이게 없으면 `설명 ㄴ`은 마법문구가 된다.
이게 있으면 `설명 ㄴ`은 외부기억에서 나온 실행 압축이 된다.

피키는 Traceability 필드를 정식 채택한다.
